## 2. Rate Limit

| 항목 | 값 | 출처 |
|---|---|---|
| 한도 | **5 calls/sec** | `config.yaml broker.rate_limit_calls`, `rate_limit_period` |
| 알고리즘 | 슬라이딩 윈도우 1초 | `core/rate_limiter.py` |
| 이론상 최소 간격 | 200ms/call | 5 calls ÷ 1000ms |
| 분당 한도 (환산) | 300 calls/min | 5 × 60 |
| 시간당 한도 | 18,000 calls/hour | 5 × 3600 |

**주의**: 키움 공식 문서의 글로벌 한도 (일별/IP별)는 별도 명시 없음. 위 값은 config 기반 내부 제한 — 실제 키움 서버는 더 관대할 가능성.

## 4. 종목당 1년치 수집 추정

### 입력
- 기간: 2025-04-01 ~ 2026-04-15 = **268 거래일**
- 일당 분봉: 300개 (실측 기준)
- 종목당 총 분봉: 268 × 300 = **80,400개**

### API 호출 수
- 페이지당 900개 → 80,400 ÷ 900 = **90회 호출/종목**

### 시간 추정

| 시나리오 | 초당 호출 | 종목당 시간 | 60종목 총 시간 |
|---|---|---|---|
| Rate limit 포화 | 5 calls/s | 90 / 5 = **18초** | 60 × 18s = **18분** |
| 실측 응답 + DB 저장 병목 | ~3 calls/s | 90 / 3 = **30초** | 60 × 30s = **30분** |
| DB per-row commit 병목 (현 구현) | ~0.3 pages/s | 90 × 3.2s = **5분** | 60 × 5min = **5시간** |

**DB 저장이 주 병목**: `data_collector.py:118` `await db.execute_safe(_INSERT_SQL, params)` — candle마다 commit. 900건/페이지 × 0.003s = 2.7초.

→ **batch INSERT + 단일 commit으로 최적화 시 30분 이내 가능**

## 8. 권장 전략

### A안: 현 구현 그대로 실행
- 5~6시간 소요 (밤새 백그라운드)
- 코드 수정 0건
- 위험: 네트워크/세션 끊김 시 재시작

### B안: DB 최적화 후 실행 (권장)
- `DataCollector._parse_and_save`를 `executemany` + 단일 commit으로 변경
- 30~40분 소요
- 코드 수정 ≈ 10줄

### C안: 병렬 (금지)
- Rate limit 5/sec가 글로벌 → asyncio 동시성 늘려도 한도 동일
- 이득 없음

## 9. 발견된 제약

1. **페이지네이션 종료 조건**: API가 과거 데이터 없을 때 같은 `cntr_tm`을 반복. `data_collector.py:73-76`에서 `base_dt` 중복 감지로 처리.
2. **응답 시간 변동성**: 첫 호출 0.2초, 재호출 0.036초 — 세션/TCP 캐시 효과.
3. **분봉 실제 개수**: 일당 300개 (점심 휴장 11:30~13:00 반영됐을 가능성). 기존 `CANDLES_PER_DAY=400` 추정이 과다.
