# ADR-004: 블랙리스트 집계 단위 재검증 — baseline 일치 확인

## 일자
2026-04-15

## 결정
라이브 `risk_manager.is_ticker_blacklisted(ticker)` 와 백테스터 `run_multi_day_cached`의 블랙리스트 로직은 **모두 ticker-scoped**로 등가. 추가 수정 없이 **현 상태 유지**.

## 배경
Phase 2 단계 2 대조 문서(`docs/spec/live_baseline_comparison.md:§6`)에서 "백테스트=전체 집계 vs 라이브=종목별 집계"로 기술되었으나 재검증 결과 **해당 분석이 부정확**.

### 코드 재검증

**백테스터** (`backtest/backtester.py:581-599`):
```python
for t in all_trades:  # all_trades = 현재 워커의 누적 trades
    ...
    if cutoff <= exit_date < date and t["pnl"] < 0:
        recent_losses += 1
```
`all_trades`는 `run_multi_day_cached` 함수 내 로컬 변수. `scripts/backtest_single.py:116-117`의 `ProcessPoolExecutor`가 **ticker 단위로 워커를 분산**하므로, 각 워커의 `all_trades`는 해당 ticker의 trades만 포함.

**라이브** (`risk/risk_manager.py:215-220`):
```sql
SELECT COUNT(*) FROM trades
WHERE ticker=? AND side='sell' AND pnl<0
  AND date(traded_at)>=?
```
SQL 자체가 `ticker=?`로 필터링.

### 결론
두 구현 모두 **"해당 ticker의 최근 N일 손실 거래 수"**를 집계. 판정 경로 다르지만 결과는 동일.

## 미세 차이 (수정 범위 외)
- 백테스터: `cutoff ≤ exit_date < date` — 오늘 제외 (엄밀히 [D-N, D-1])
- 라이브: `date(traded_at) >= since` — 오늘 포함 ([D-N, D])
- 백테스터는 일별 루프 시작 시에만 판정하므로 오늘 trades 자체가 아직 없음. 결과적으로 실행상 차이 거의 없음.
- 정밀 일치를 원하면 라이브 SQL에 `AND date(traded_at) < date('now','localtime')` 추가 가능 (현재 미적용 — 영향 미미).

## 대안
1. 라이브 SQL 에 오늘 제외 조건 추가 → 1줄 수정, 영향 미미
2. 현 상태 유지 → baseline 실질 등가 (선택)

## 결정 이유
- 실질 등가이며 PF 2.91 baseline 재현에 영향 없음
- 라이브의 "오늘 포함" 동작이 오히려 실시간 매매 방어 측면에선 더 보수적
- 수정 이익 대비 변경 위험 미미

## 상태
ACCEPTED — 코드 변경 없음. Phase 2 단계 2 분석 오류 정정.
